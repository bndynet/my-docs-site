# my-docs-site

Documentation site built on [Docusaurus 3](https://docusaurus.io/) + the
[`@bndynet/docs`](packages/docs) site template.

The root directory is a **consumer** of the template package: all theme
swizzles, pages, components, CSS and static assets live under
`packages/docs/assets/`. The root only contains site settings
(`docs.config.js`), content (`docs/`, `blog/`) and deployment scripts.

## Repository layout

```
.
├─ docs.config.js         # site settings (title, nav, footer, externalScripts, …)
├─ docs.config.prod.js    # production overrides (only the fields that differ)
├─ docs/                  # MDX/MD content served at /
├─ blog/                  # optional blog posts
├─ packages/
│  └─ docs/               # @bndynet/docs — the site template package
│     ├─ bin/docs.cjs     # CLI
│     ├─ lib/             # config/plugin glue (Docusaurus configfactory)
│     └─ assets/          # shipped template (src/components, src/theme,
│                         #   src/pages, src/utils, src/css, static/)
```

## Daily workflow

### Install (once)

```bash
npm install
```

This is an npm workspaces monorepo: the root site and `packages/docs`
share one `node_modules`. `@bndynet/docs` is symlinked automatically.

### Start the dev server

```bash
npm start          # runs `docs dev` against docs.config.js
```

### Edit content

Content lives under `docs/` and `blog/` — edit any `.md`/`.mdx` file and
the dev server hot-reloads.

### Customize the template

All template files live in **`packages/docs/assets/`**. Edit them in place
— the dev server picks up changes immediately through workspace symlinks.

- Components → `packages/docs/assets/src/components/`
- Swizzles   → `packages/docs/assets/src/theme/`
- Pages      → `packages/docs/assets/src/pages/`
- CSS        → `packages/docs/assets/src/css/custom.scss`
- Static     → `packages/docs/assets/static/`

To override a single file per-site (without modifying the package), place
a file with the same path under the root's `src/…` or `static/…` — the
package's Webpack resolver prefers the consumer copy.

### Build

```bash
npm run build          # baseUrl: '/' (default)
npm run build:gh       # baseUrl: '/my-docs-site/' (docs.config.prod.js)
```

Only fields that differ from the base config go into `docs.config.prod.js`:

```js
// docs.config.prod.js
const base = require('./docs.config.js');
module.exports = { ...base, baseUrl: '/my-docs-site/' };
```

`/tw.config.js` and other baseUrl-relative `externalScripts` are auto-prefixed
for you — you only declare them once in `docs.config.js`.

### Deploy (GitHub Pages)

```bash
USE_SSH=true npm run deploy
# or
GIT_USER=<GitHub username> npm run deploy
```

## Adding a new docs section

1. Create a folder under `docs/`, e.g. `docs/reference/`.
2. Add one entry to `docs.config.js.nav`:
   ```js
   { label: 'Reference', docsPath: 'reference' }
   ```
3. Restart the dev server. `@bndynet/docs` generates an autogenerated
   sidebar scoped to `docs/reference/` keyed by the label.

## Contributing

Everything that is not site settings or content lives in `packages/docs/`.
The map below tells you **which file to edit** for each common change.

### Where to put what

| I want to…                           | Edit                                                                 |
|--------------------------------------|----------------------------------------------------------------------|
| Swizzle a `@theme/*` component       | `packages/docs/assets/src/theme/<Name>/index.(tsx\|jsx\|js)` — auto-wired via alias |
| Add a reusable React component       | `packages/docs/assets/src/components/<Name>/` — import as `@site/src/components/<Name>` |
| Add a custom page                    | `packages/docs/assets/src/pages/<route>.(js\|mdx)`                    |
| Tweak global styles                  | `packages/docs/assets/src/css/custom.scss`                           |
| Ship a static asset (logo, script…)  | `packages/docs/assets/static/`                                       |
| Add a Docusaurus plugin/preset       | `packages/docs/lib/buildDocusaurusConfig.cjs` + `peerDependencies` in `packages/docs/package.json` |
| Add a new `DocsConfig` field         | `packages/docs/lib/types.d.ts` → `packages/docs/lib/mapDocsConfig.cjs` → `packages/docs/lib/buildDocusaurusConfig.cjs` |
| Change CLI behavior (flags, init, …) | `packages/docs/bin/docs.cjs`                                         |

All of the `assets/` files can be overridden by a consumer at their own
siteDir with a same-path copy — the Webpack resolver prefers the consumer
copy (see `DocsResolvePlugin` in `buildDocusaurusConfig.cjs`).

### Upgrading Docusaurus

Docusaurus versions are pinned in **two** places and must stay in lockstep
— otherwise React context breaks at SSR (`useColorMode is called outside
<ColorModeProvider>` and friends).

1. `package.json` (root) — `dependencies` **and** `overrides`:
   ```json
   "dependencies": {
     "@docusaurus/core": "3.10.0",
     "@docusaurus/preset-classic": "3.10.0",
     "@docusaurus/theme-live-codeblock": "3.10.0",
     "@docusaurus/theme-mermaid": "3.10.0"
   },
   "overrides": {
     "webpack": "5.105.4",
     "@docusaurus/core": "3.10.0",
     "@docusaurus/theme-common": "3.10.0",
     "@docusaurus/theme-classic": "3.10.0"
   }
   ```
   The `overrides` block forces transitive deps (e.g. `theme-common` pulled
   in by a plugin on an older pin) onto the same version. Keep `webpack` at
   `5.105.x` for now: `5.106+` tightens `ProgressPlugin` validation and breaks
   the Docusaurus bundler until upstream adjusts.
2. `packages/docs/package.json` — `peerDependencies` are `*`, so nothing to
   bump there unless you want to declare a new *minimum* version.

Then from the repo root:

```bash
rm -rf node_modules package-lock.json packages/docs/node_modules
npm install
npm run build && npm run build:gh
```

The `docs` CLI runs a **dedup check** at the start of every `dev`/`build`
and prints a targeted warning if it finds more than one copy of
`@docusaurus/core`, `@docusaurus/theme-common`, or `@docusaurus/theme-classic`
— if you see that warning, fix `overrides` before doing anything else.

Also scan the [Docusaurus release notes](https://docusaurus.io/changelog)
for breaking changes to the **swizzle API** — our swizzles live under
`packages/docs/assets/src/theme/` and must match the new `@theme/*`
signatures.

### Adding a Docusaurus plugin or preset

1. Install at the **root**:
   ```bash
   npm install --save docusaurus-plugin-foo
   ```
2. Register in `packages/docs/lib/buildDocusaurusConfig.cjs` (append to the
   `plugins` array, or to `presets[0][1]` for preset-classic options).
3. Declare it in `packages/docs/package.json` under `peerDependencies`
   (and `peerDependenciesMeta` with `"optional": true` if it's not
   universally required), so downstream consumers get a clear install hint.

### Adding a new `DocsConfig` field

Each new top-level field travels through three files:

1. **Contract** — add the field to the `DocsConfig` interface in
   `packages/docs/lib/types.d.ts` with a JSDoc comment.
2. **Normalize** — `packages/docs/lib/mapDocsConfig.cjs` reads the raw
   `site` object and produces the internal `app` shape used by the builder.
   Add mapping here if the field needs defaulting or transformation.
3. **Wire** — `packages/docs/lib/buildDocusaurusConfig.cjs` consumes either
   `app.*` or the raw `site.*` and hands it to Docusaurus. Expose it through
   `customFields` if a React component needs to read it at runtime
   (e.g. `docsRouteBasePath` or the normalized `hero` block).

### Testing a change

There is no test suite yet; the smoke flow is:

```bash
npm run build        # baseUrl: '/'
npm run build:gh     # baseUrl: '/my-docs-site/', via docs.config.prod.js
```

Both builds must succeed with no dedup warnings. For UI-touching changes,
also run `npm start` and click through the home page, a docs page with a
sidebar, the `/iframe` route, and dark/light toggle.

### Publishing `@bndynet/docs`

1. Bump `packages/docs/package.json`'s `version` (semver: **major** if you
   changed `DocsConfig`, the CLI flag set, or any consumer-visible
   `@site/*` path; **minor** for additive features; **patch** for fixes).
2. Update `packages/docs/README.md` if the public surface changed.
3. Publish:
   ```bash
   cd packages/docs && npm publish --access public
   ```
4. Consumers pick up the change on their next `npm install`.

## Troubleshooting

- **`Cannot read properties of undefined (reading 'mdxCrossCompilerCache')`** —
  two copies of `@docusaurus/*` got installed. Run `npm dedupe` or verify
  the root `overrides` pins them all to the same version.
- **`useColorMode is called outside <ColorModeProvider>`** — same root cause
  as above: multiple `@docusaurus/theme-common` copies. Deduplicate.
- **`Cannot find @docusaurus/core`** — CLI couldn't locate Docusaurus. Make
  sure it's in the root's `dependencies` (or install at the siteDir).
